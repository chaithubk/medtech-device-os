## Workflow Overview

The CI pipeline is defined in `.github/workflows/device-build-smart.yml`.

It runs on GitHub-hosted Ubuntu 22.04 runners with a 360-minute job timeout.

**What CI does on every run:**
1. Frees up disk space on the runner (~12–15 GB freed)
2. Installs Yocto host tools
3. Clones Yocto layers (poky, meta-openembedded, meta-qt6)
4. Initializes the build directory from sample configs
5. Runs Yocto preflight checks (metadata parse + URI validation)
6. Builds the image with `bitbake`
7. Runs post-build policy checks (no debug packages, Python sanity)
8. Packages the release bundle
9. Verifies the release bundle
10. Uploads artifacts to GitHub Actions

**Additionally on `main` branch push or `workflow_dispatch`:**
11. Creates a GitHub Release with the QEMU bundle

## Job Steps Explained

### Free up disk space

GitHub-hosted runners have ~30 GB total disk. The Yocto build needs ~20+ GB.
This step removes unused tools (dotnet, Android SDK, Haskell, Go, Node, Ruby,
Python, CodeQL) and Docker images to reclaim ~12–15 GB.

### Set up build environment

Installs the Yocto host dependencies via `apt-get`. These are the standard
Ubuntu packages required by Poky/kirkstone.

### Clone Yocto layers

Clones three layers with `--depth 1` (shallow) to minimize download time:
- `poky` (kirkstone) — Yocto/OE build system
- `meta-openembedded` (kirkstone) — Extended package set
- `meta-qt6` (6.4) — Qt6 framework

For `meta-openembedded`, only the three required sublayers are kept
(`meta-oe`, `meta-python`, `meta-networking`); others are deleted to save space.

### Initialize build directory

Sources `oe-init-build-env` to create the `yocto/build/` directory, then
copies the sample config files:
- `yocto/conf/local.conf.sample` → `yocto/build/conf/local.conf`
- `yocto/conf/bblayers.conf.sample` → `yocto/build/conf/bblayers.conf`

These sample files are tracked in git and define the canonical CI configuration.

### Pre-build disk audit

Reports available disk space before the build starts. Fails early if less than
5 GB is available (the build would fail anyway).

### Yocto preflight

Two fast checks before the full build:

1. **`bitbake -p`** — Parse all metadata. Catches layer configuration errors,
   missing providers, and recipe syntax errors early (takes ~2 minutes).

2. **`bitbake -k ... -c checkuri`** — Validates SRC_URI for custom recipes
   without downloading. Catches broken URLs and incorrect checksums.

### Build CI image

```bash
bitbake core-image-medtech
```

This is the main build step. Typical duration on GitHub runners: 60–120 minutes
(no sstate cache between runs).

### Post-build checks

1. **Verify image was built** — Confirms the `.ext4` file exists and has a
   reasonable size.

2. **Enforce no debug/dev packages** — Scans the package manifest for any
   `-dbg`, `-dev`, `-staticdev` packages and fails if found.

3. **Python runtime sanity** — Runs `scripts/verify-image.sh python-sanity`
   to confirm Python packages are correctly installed.

4. **Process SPDX SBOM** — Collects SPDX SBOM files generated by BitBake.

### Package release bundle

```bash
bash scripts/package-release-artifacts.sh --image-name core-image-medtech
```

Produces in `artifacts/`:
- `core-image-medtech-qemuarm64-bundle.tar.gz` — kernel + rootfs + DTBs
- `core-image-medtech-qemuarm64-manifest.json` — build metadata + checksums
- `SHA256SUMS` — checksum file for the above two

### Verify release bundle

```bash
bash scripts/verify-release-package.sh --image-name core-image-medtech
```

Verifies SHA256 checksums and confirms the archive contains required files.

## Build Outputs

### GitHub Actions artifacts

Each run uploads to GitHub Actions (retained 30 days):

| Artifact | Contents |
|---|---|
| `qemu-image-medtech` | `artifacts/` directory with bundle, manifest, SHA256SUMS |
| `yocto-debug-failed-recipe` | Task logs for failed recipes (failure runs only) |

### GitHub Releases (main branch only)

| Asset | Description |
|---|---|
| `core-image-medtech-qemuarm64-bundle.tar.gz` | QEMU-bootable image bundle |
| `core-image-medtech-qemuarm64-manifest.json` | Build metadata |
| `SHA256SUMS` | Integrity checksums |

## Release Creation

Releases are created automatically:

**Push to `main`:**
- Tag: `latest` (replaces the previous `latest` release)
- Marked as latest release (not pre-release)
- Suitable for end users

**`workflow_dispatch` (manual trigger):**
- Tag: `dev-<full-sha>` (unique, never overwrites)
- Marked as pre-release
- Suitable for testing release candidates

The release body includes:
- Commit SHA and branch name
- Quick start command (one-liner)
- Checksum verification command

## Failure Handling

When a build step fails:

1. **Dump failed BitBake task logs** — Prints the last 120 lines of failed
   task log files and the `config.log` for autoconf failures.

2. **Collect failed recipe debug logs** — Copies task logs, run scripts, and
   the full `bitbake -e` environment dump to `artifacts-debug/`.

3. **Upload failed recipe debug logs** — Uploads `artifacts-debug/` as a
   GitHub Actions artifact for post-mortem analysis.

## CI vs Local Builds

### Files that affect CI behavior

| File | Purpose |
|---|---|
| `.github/workflows/device-build-smart.yml` | Job definition and all build steps |
| `yocto/conf/bblayers.conf.sample` | Layer list for CI (copied to build dir) |
| `yocto/conf/local.conf.sample` | Build config for CI (copied to build dir) |
| `yocto/meta-medtech/**` | Custom recipes and layer |

### Files that are local/dev-container only

| File | Purpose |
|---|---|
| `.devcontainer/Dockerfile` | Container image definition |
| `.devcontainer/devcontainer.json` | VS Code dev container settings |
| `scripts/quick-setup.sh` | Local layer clone and setup |
| `scripts/setup-devenv.sh` | Extended local environment setup |
| `yocto/build/conf/bblayers.conf` | Generated per-machine config (gitignored) |
| `yocto/build/conf/local.conf` | Generated per-machine config (gitignored) |

## Modifying the Pipeline

### Adding a new build step

Add a step to `.github/workflows/device-build-smart.yml`:

```yaml
- name: My new step
  run: |
    echo "Doing something"
    bash scripts/my-new-script.sh
```

### Changing the build image

Update `CI_BUILD_IMAGE` at the top of the workflow:

```yaml
env:
  CI_BUILD_IMAGE: core-image-medtech  # change this
```

### Adding a new layer dependency

1. Add a clone step to the workflow
2. Update `yocto/conf/bblayers.conf.sample` to include the new layer
3. Update `yocto/meta-medtech/conf/layer.conf` `LAYERDEPENDS` if needed

### Extending release notes

Edit the `body` section in the "Create GitHub Release with Artifacts" step.
